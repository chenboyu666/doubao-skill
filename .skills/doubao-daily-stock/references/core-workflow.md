## 适用场景

当用户输入股票名称、股票代码、证券简称，或提出任何围绕单只股票的分析问题时，使用本 skill。问题可以是标准每日复盘，也可以是更个性化的目标价、短线交易、事件催化、持仓处理、估值贵不贵、风险是否变大等。

典型触发问题：

- “帮我分析下某某公司 / 生成某股每日复盘 / 做一份飞书文档版个股分析。”
- “我知道英伟达是好公司，但股价还能到什么位置？”
- “我喜欢短线操作，中船气体现在能不能入手快赚一笔？”
- “中通客车 5 月销量增长，这对公司股价会有拉升吗？”
- “我已经持仓/空仓，今天该拿、减、等还是做短线？”
- “这个公告、订单、业绩、行业新闻对公司股价影响大不大？”

目标是端到端完成：

1. 识别股票和用户真正关心的问题。
2. 获取行情、资金、基本面、消息、技术和预期数据。
3. 形成机构风格、但能直击用户问题的个股分析。
4. 默认先输出对话版完整日报（日报内最后一句为“如果你需要，我可以把以上内容写入一份飞书文档。”）；用户确认后再生成飞书文档版完整报告。
5. 用户确认后，调用 `lark-doc` skill 新建或编辑飞书文档；不得只在聊天界面重新输出完整报告内容。
6. 文档标题、摘要指标表、正文段落、表格、信息来源和免责声明必须稳定复现。

## 规范加载

本 skill 是标准多文件目录。执行时先读取目录根部 `SKILL.md`，再按文件导航在需要时读取 `references/` 与 `scripts/` 中的具体文件。不要把规则自由改写成另一套版式。

## 总工作流

### 1. 股票识别与用户问题识别

先同时识别两件事：

- 股票主体：股票名称、证券代码、市场、币种。
- 用户问题：每日复盘、目标价/上涨空间、短线交易、事件催化、持仓处理、风险判断、估值贵不贵等。

必须把用户问题转化成 `question_focus`，并贯穿对话版日报和后续飞书文档版正文。不要只生成模板化日报而不回答用户原问题。

常见 `question_focus`：

- `target_price`：用户关心股价能到哪里。重点看技术压力位、震荡区间、投行一致预期、估值分位、业绩弹性，给出情景区间而非确定目标价。
- `short_trade`：用户想短线快进快出。重点看今日量价、资金流、可核验技术指标、支撑压力和波动风险；只能给研究视角的短线环境判断，不给具体买卖、仓位、止损或止盈指令。
- `event_catalyst`：用户问销量、订单、公告、政策、行业新闻是否会拉动股价。重点核查事件真实性、收入/利润贡献、持续性、是否已被股价反映，并说明对短期情绪和中期基本面的不同影响。
- `position_advice`：用户提到持仓、空仓、被套、盈利。重点分持仓状态给出操作边界和风控。
- `daily_recap`：用户只要求复盘。按标准日报生成，但仍要提炼最重要矛盾。

如果用户问题不清楚，先用最接近的 `question_focus` 推断；只有股票主体无法确定时才向用户确认。

- A股：支持 `300308`、`SZ300308`、`300308.SZ`、`中际旭创` 等。
- 港股：支持 `00700`、`0700.HK`、`腾讯控股` 等。
- 美股：支持 `NVDA`、`MSFT` 等。

如果名称可能对应多只股票，先用搜索或行情接口 disambiguate；无法确定时向用户确认。

若同一公司同时有 A 股和港股上市主体，且用户只给公司简称、没有明确代码或市场，例如“紫金矿业”“中国平安”“比亚迪”，必须先询问用户要分析 A 股还是港股；在用户确认前不得取数和写报告。若用户同时给出 A/H 两个代码但没有说明主分析对象，先让用户选择“以 A 股为主 / 以港股为主 / 做 A/H 对比”。本 skill 默认只分析一个主交易标的；A/H 对比需要用户明确要求。

### 2. 数据获取

必须获取或尽力获取以下数据。近期、实时、行情、新闻、预测都属于高时效内容，必须联网或调用可用行情/数据工具，不要只凭记忆。

数据验收先行硬规则：

- 本报告的数据准确性高于生成速度和页面完整度。任何关键数据只要日期、市场、粒度、单位或来源有误，后续分析都必须视为无效。
- 在写任何分析段落前，先确定 `report_request_time`（用户发起请求时间）、`data_as_of`（行情截至时间）和最近有效交易日。若请求发生在周末、节假日、休市后或停牌期间，应按交易所日历向前推到最近一个有真实交易的交易日；如果当日盘中请求，则以请求时点可获得的最新行情为准。
- 先纵览报告需要的全部数据字段，再统一取数和核验。不要边写报告边临时补数据，除非发现必要缺口；补充数据也必须先核验。
- 必须先形成内部数据核验记录。该记录不要求保存为本地 `.txt`，也不得作为报告信息来源。记录至少包括：股票身份、请求时间、最近有效交易日、数据截至时间、每个字段的值、单位、来源名称、可打开原始外部链接或 `seed_finance_search（同花顺数据库）` 工具来源标识、来源权威性判断、缺失原因、自行计算公式、新闻/公告原文链接、最终是否允许进入报告。
- 后续对话版日报和飞书文档版报告原则上只能使用已核验数据。内部记录中标记为“未核验/冲突/可疑/缺失”的数据，不得进入正文判断、表格或技术分析。
- 当前版本采用“权威单源 + 清晰口径”验收。金融数据若由 `seed_finance_search` 明确返回，且证券代码、交易市场、交易日期/时间、字段含义、复权口径和单位清楚，可以直接采用；非强制专用数据未返回或口径不清时，使用 `general_search` 查找官方或权威机构来源作为兜底。摘要指标 8 项和 A 股资金流向指标只能来自 `seed_finance_search（同花顺数据库）`，不得使用 `general_search` 或其他来源兜底。港股/美股资金净流入/净流出、港股通南向资金若 seed 未返回，可用 `general_search` 查找官方或权威机构来源，但数据日期必须是报告当日或最近一个有效交易日。

取数入口硬规则：

- 在豆包等支持专用搜索工具的环境中，主要调用 `seed_finance_search` 与 `general_search`。
- `seed_finance_search` 是金融领域数据查询的第一入口，优先用于证券识别、实时/收盘行情、成交量、PE TTM、A 股资金流、板块、技术指标、一致预期、上市公司财务数据、汇率、基金、期货、贵金属和常见宏观数据。量比不直接取数，只能在收盘口径下使用 seed 返回的成交量自行计算；盘中口径写 `暂无`。
- `general_search` 主要用于非查数类需求，例如公司公告、交易所/监管披露、权威财经新闻、产业媒体、行业事件、公司业务介绍、分析材料和信息来源原文；当非强制专用数据在 `seed_finance_search` 搜不到、字段缺失或口径不清时，也用 `general_search` 检索官方或权威机构来源兜底。
- 不论来源入口如何，都必须先通过数据验收，再进入正文分析；任何经 `general_search` 采用的数据或事实必须来自官方或权威机构来源，不得采用雪球、股吧、贴吧、个人用户评论或无法确认原始来源的内容。

数据来源前置标准化硬规则：

- 当前版本不获取 1 分钟级行情，不获取 30 个交易日日线，不获取 90 个交易日行情。
- 所有进入报告的数值必须先进入内部数据核验记录，并记录为统一结构：`字段名 / 数值 / 单位 / 时间口径 / 来源名称 / 原始外部来源链接或工具来源标识 / 是否核验 / 是否进入报告`。
- 行情摘要 8 项和 A 股资金流向指标必须使用 `seed_finance_search（同花顺数据库）` 直接披露值或基于其返回值自行计算；不得用其他来源替代。量比只在收盘口径下自行计算，公式为 `当日成交量 / 过去 5 个交易日平均成交量`，输入成交量必须来自 `seed_finance_search（同花顺数据库）`；盘中口径写 `暂无`。港股/美股资金净流入/净流出、港股通南向资金先查 seed，未返回时可用 `general_search` 检索官方或权威机构来源，但数据日期必须是报告当日或最近一个有效交易日。板块行情、MA、MACD、RSI、投行一致预期等其他数据优先使用数据源直接披露值。
- 若数据由 agent 自行计算，只能作为兜底，并必须写入内部数据核验记录：输入字段、输入来源链接、公式、计算过程、计算结果和“自行计算”标记。飞书文档版或对话版中引用时仍要给原始输入来源编号。
- 凡找不到可打开来源链接或 `seed_finance_search（同花顺数据库）` 工具来源标识、无法核验、日期/市场/单位冲突或口径不明的数据，不得进入正文、表格、综合判断或风险判断。

必须获取：

- 摘要指标：当前价格/收盘价、涨跌幅、日内区间、总市值、市盈率 TTM、成交额、换手率直接来自 `seed_finance_search（同花顺数据库）`；量比在收盘口径下使用同一来源返回的当日成交量和过去 5 个交易日成交量自行计算，盘中口径写 `暂无`。
- A 股资金流：主力资金、超大单资金、大单资金的流入/流出情况；陆港通标的追加北向资金。资金字段只能来自 `seed_finance_search（同花顺数据库）`。中单和小单暂不展示。
- 港股资金流：资金净流入/净流出；港股通标的在有权威来源时追加南向资金净流入/净流出。港股不分大单小单，不生成 A 股“主力资金 / 超大单 / 大单 / 北向资金”表。资金字段先查 `seed_finance_search`，若未返回，可用 `general_search` 检索官方或权威机构来源，但数据日期必须是报告当日或最近一个有效交易日。
- 美股资金流：资金净流入/净流出。美股生成一行资金流表，不再用成交量、期权、ETF 或机构持仓替代资金流指标。资金字段先查 `seed_finance_search`，若未返回，可用 `general_search` 检索官方或权威机构来源，但数据日期必须是报告当日或最近一个有效交易日。
- 相关板块：最相关的 1-3 个行业/概念/主题板块，当日涨跌、涨跌家数、龙头/权重地位。
- 技术指标：MA、MACD、RSI，以及震荡区间、压力位/压力区间、支撑位/支撑区间。
- 消息：最近 3 日内影响公司业务和股价的权威财经新闻、产业新闻、公司公告、交易所/监管披露、行业事件；未来当天剩余时间和未来几日的潜在催化。
- 投行一致预期：A 股或有完整预测时获取未来 3 年归母净利润、营业收入、EPS、PE；港股、美股或主要披露券商评级/目标价时，获取最近约 5 家机构的机构名称、最新评级、目标价、日期。
- 风险：当前走势风险、估值风险、业绩/业务风险、流动性和情绪风险。

按 `question_focus` 补充取数：

- `target_price`：补充一致预期 EPS/PE、历史估值区间、可核验压力位/支撑位、上方缺口/前高/整数位、可比公司估值。
- `short_trade`：补充当日收盘/盘中快照、成交额相对 5 日均值、短线资金流和可核验关键波动区间，用于说明风险环境，不用于生成具体交易指令。
- `event_catalyst`：补充事件公告/新闻原文、事件发生日期、市场是否已交易该事件、对收入/利润/订单/产能/政策预期的可能影响。
- `position_advice`：补充成本区间无法获知时，用当前价上下关键位给出通用持仓/空仓/短线三类建议。

详见本文后部“数据字段与计算口径”。

技术指标核验：

- MA、MACD、RSI、换手率等技术/交易指标必须优先取自可打开、可核验的数据源直接披露值，并在内部数据核验记录中记录来源链接或工具来源标识。量比收盘后自行计算，并记录当日成交量、过去 5 个交易日成交量、公式和输入来源；盘中口径记录缺失原因并展示 `暂无`。
- 技术指标若由 `seed_finance_search` 明确返回，且参数、时间口径和单位清楚，可以直接展示；若未返回，用 `general_search` 查找官方或权威技术指标/行情数据源。若来源口径不清，不展示具体读数，只保留审慎文字判断或写“技术指标暂未核验”。
- 自行计算只可作为兜底；必须披露公式、输入数据来源、参数和“自行计算”标记，不得把自行计算结果伪装成平台披露。

新闻、公告、研报与一致预期核验：

- 新闻和公告必须核查最近 3 日内的原文、发布日期、来源机构和是否与该公司/行业直接相关；过早、无原文、无权威来源或只讨论资金异动的内容，不得作为第四部分“消息面复盘”的主依据。
- 研究报告和一致预期先用 `seed_finance_search` 查询结构化金融数据；未返回时，用 `general_search` 查找可打开的券商研报摘要、评级页面、机构预测页面、公司/交易所披露或权威数据库页面。若只取得 EPS、目标价或评级等部分字段，必须说明表格口径和自行计算项；表格下方保留简短口径说明。

### 3. 分析写作

报告要像交易台盘后快报，不要像泛泛新闻摘要。

写作要求：

- “今日行情、量能与资金流向”是开头核心段，必须把价格走势和成交量、换手率、量比、资金流结合解释，并进一步回答“为什么会这样走”。价格和量能是结果，必须尝试追溯到公司基本面、行业景气、客户/产品/订单、政策/宏观、海外映射、板块风险偏好或重要公告等原因。若没有检索到明确原因，必须写明“未见明确基本面催化，盘面更偏交易结构/板块情绪驱动”，不得臆造。
- 若用户提出个性化问题，第一部分必须明确说明今日盘面是否支持用户的判断：例如目标价问题要说明当前量价是否支持继续上攻；短线问题要说明追高、回踩或突破确认哪个更合适；事件催化问题要说明盘面是否已经反映该事件。
- “板块联动与个股地位”要说明个股相对板块是否强势、是否带动板块、是否只是跟随 beta。
- “技术面分析”要把 MA、MACD、RSI 分别拆成：短标题、具体内容、状态标签；关键价位要落到区间带图和表格。具体内容必须照顾不懂技术指标的读者，用 1-2 句完整中文解释“这个指标现在说明什么”，不要只堆 MA5/MA10/DIF/DEA/RSI 数字。
- “消息面与隔夜观察”必须引用最近 3 日内的权威财经媒体、产业媒体、公司公告/IR、交易所/监管披露等来源，聚焦公司基本面、产业基本面、行业趋势和未来催化；不要把成交额、大宗交易、龙虎榜、主力净流入等资金异动新闻重复写成消息面。
- “综合评估”要给出研究评估和充分评估理由，必须包含基本面或行业趋势判断，并结合估值、技术、资金或消息面证据；不得输出空仓者、持仓者、短线参与者等具体操作建议表格。
- 第六部分必须直接回答 `question_focus`：目标价问题给情景区间与失效条件；短线问题只说明短线交易环境、波动和风险，不给具体买卖/仓位/止损/止盈指令；事件催化问题给短期情绪影响与中期基本面影响；持仓问题只能给研究视角的风险边界，不给个性化交易指令。
- “短期风险提示”要对散户友好，重点提醒追涨杀跌、估值高位、消息不及预期、客户/行业/资金面变化。

### 4. 对话版完整日报输出

由于完整飞书文档版生成和复核耗时较长，调用本 skill 时默认先输出“对话版完整日报”，不直接生成飞书文档版完整报告。对话版日报必须承载完整观点和分析深度，不能只给 3-5 条简短摘要。

触发时机：

- 完成核心数据抓取、校验和分析后立即输出。
- 核心数据至少包括：当前价/涨跌幅、成交额/换手率/量比、按市场口径列示的资金流、最相关板块表现、关键技术位、最近 3 日基本面/产业面消息或“未检索到权威消息”、一致预期或缺失说明。其中当前价/涨跌幅/成交额/换手率必须来自 `seed_finance_search（同花顺数据库）`；A 股资金流必须来自 seed；港股/美股资金净流入/净流出可在 seed 缺失时使用 `general_search` 权威来源，但必须是报告当日或最近一个有效交易日数据；量比必须基于 seed 的收盘成交量自行计算，盘中写 `暂无`。
- 若个别数据缺失，必须在正文中标注缺失项、替代来源或估算口径，不得编造。

内容要求：

- 对话版正文最开头必须先输出固定提示语：`「以下内容仅为行情与逻辑分析，不构成任何投资买卖建议，务必结合自身风险承受能力决策」`；该提示语位于报告标题之前，且只出现一次。
- 标题固定为 `个股日报 / 对话版深度复盘`。
- 开头展示 `核心结论`，列 3-5 点，第一点直接回答用户问题；但实际撰写顺序放到最后，必须等全部数据、资讯和分项分析完成后再回写开头结论。
- 正文必须按以下固定结构输出，不得只给摘要：
  1. `一、今日行情、量能与资金流向`
  2. `二、板块联动与个股地位`
  3. `三、技术面分析`
  4. `四、消息面与隔夜观察`
  5. `五、投行一致预期`
  6. `六、综合评估`
  7. `七、短期风险提示`
  8. `信息来源`
  9. `免责声明`
- 每个主体章节至少 1 段深度分析；第一、三、四、六部分应更充分，避免空泛。
- 不要把 8 个摘要指标逐项重复一遍；必须把数据转化为判断。
- 不要使用“必涨、稳赚、满仓”等绝对化语言。
- 不得以报告自身口吻输出直接投资操作建议，例如“建议买入 / 卖出 / 加仓 / 减仓”“适合买入”“应减仓”等。允许在客观转述、条件假设、风险提示或用户既有状态描述中出现相关词汇，但必须明确保持研究分析口径，不得形成操作指令或替用户作出交易决策。
- 正负颜色语义仍按 A 股习惯：涨/净流入/正数为红，跌/净流出/负数为绿。
- 若用户问“股价能到什么位置”，核心结论第一条必须写成情景判断：上方第一观察区间、乐观情景触发条件、失效条件，不能给确定承诺。
- 若用户问“短线能不能快赚”，核心结论第一条必须从研究合规角度说明当前短线波动环境和风险，不得给具体买卖、仓位、止损或止盈指令。
- 若用户问“某事件会不会拉股价”，核心结论第一条必须先回答“短期情绪催化/中期基本面支撑/影响有限/已被定价”，并写明事实依据。
- 不要自动开始生成飞书文档版。

聊天渲染样式执行：

- 完成取数/搜索与内部数据核验记录后、生成完整 Markdown 内容底稿前，必须读取 [content-rules.md](content-rules.md)。该文件负责内容写作：各章节写什么、如何分析、哪些事实和判断可以进入报告；不得在取数前提前读取该文件。
- 完整 Markdown 内容底稿必须先独立完成；底稿完成后、输出聊天渲染结果前，再读取 [dialogue-style.md](dialogue-style.md)。该文件只负责聊天界面渲染：标题结构、章节空行、段落空行、重点加粗、表格居中、推荐结构和展示检查，不得干扰 Markdown 内容底稿的段落风格。
- 输出聊天渲染结果前，必须按 [dialogue-style.md](dialogue-style.md) 完成展示检查并修正；不得借展示调优删减、压缩或重写内容。

如果用户已经在本轮明确要求“直接生成飞书文档版”，也必须先给简短状态说明和对话版核心结论，再进入飞书文档版生成；但不要省略数据校验、来源核验和复核。

### 5. 用户确认后生成飞书文档版

只有当用户明确表示“生成飞书文档版 / 做成飞书文档 / 需要完整报告 / 做成文档 / 继续生成”时，才进入 `lark-doc` 飞书文档编辑流程：

- 进入飞书文档编辑流程后的第一步，必须从头到尾完整读取 [feishu-doc-style.md](feishu-doc-style.md)；不得只搜索关键词、只读取局部片段、只依赖 `SKILL.md` 或本文摘要生成飞书文档。
- 完整读取 [feishu-doc-style.md](feishu-doc-style.md) 后，再基于同一份 Markdown 内容底稿组装 `lark-doc` XML；不得在飞书文档阶段重新生成事实、观点、章节或来源编号。
- 生成飞书文档版时必须先严格按照 [feishu-doc-style.md](feishu-doc-style.md) 的固定结构、数据契约、章节顺序、引用规则和 QA 清单生成 `lark-doc` 可写入内容。
- 必须调用 `lark-doc` skill 新建或编辑飞书文档。若用户已提供目标飞书文档，优先写入该文档；若没有目标文档，先询问用户提供链接/授权或确认新建文档。
- 不得用“在聊天界面输出完整飞书文档版正文”代替写入飞书文档。只有 `lark-doc` 不可用且用户明确同意看草稿时，才输出完整标签版内容。
- 飞书文档强调色目标使用 `#005689`；若 `lark-doc` 只支持命名色，则用 `text-color="orange"` 作为 fallback。一级章节用 `h1`，副标题用 `h3`，表格表头居中；正文单元格 10 个字符以内居中，长解释左对齐。
- 若目标文档可能已有评论、图片、附件或用户手工编辑内容，全文覆盖写入前必须先确认；未确认时只允许读取结构后局部替换或追加。
- 飞书文档版开头和对话版报告必须使用同一个生成时间：用户发起请求的 `report_request_time`。不得用文件写入时间、agent 完成时间或浏览器打开时间替代。
- 只使用 `lark-doc` 支持的结构化标签。
- 首屏内部顺序必须固定：文档标题 -> 短免责声明提示 -> 生成日期/数据截至/报告口径/公司简介 -> 核心结论 -> 摘要指标表 -> 第一部分正文。
- 摘要指标使用 4 行 4 列 `lark-doc` 表格：表头为“指标 / 数值 / 指标 / 数值”，并用 `<colgroup>` 控制列宽。单位直接写进数值，每个数值带来源编号。
- 摘要指标表下方必须有固定注释，说明 8 个行情摘要指标的时间口径，例如：`注：以上行情摘要截至 2026-06-12 13:30（交易日盘中）。` 或 `注：以上行情摘要为 2026-06-12 收盘口径。`
- 主章节顺序固定：
  1. 今日行情、量能与资金流向
  2. 板块联动与个股地位
  3. 技术面分析
  4. 消息面与隔夜观察
  5. 投行一致预期
  6. 综合评估
  7. 短期风险提示
- 不新增章节破坏固定版式；用户个性化问题必须嵌入顶部公司简介/开篇判断、第一部分“价格与量能”、第三部分“技术面分析”、第四部分“消息面与隔夜观察”、第六部分“综合评估”和第七部分“短期风险提示”。

技术指标、压力位、支撑位和价格区间必须来自可核验数据源披露值、研报/公告/行情页，或带公式说明的自行计算。

### 6. 输出

第一轮默认输出：

- 直接输出“对话版完整日报”的聊天渲染结果。
- 不创建飞书文档版文件，除非用户已经明确确认需要。

用户确认生成飞书文档版后，优先通过 `lark-doc` 交付：

- 已提供目标文档：使用 `lark-doc` 写入或替换该文档内容。
- 未提供目标文档：询问用户提供飞书文档链接/授权，或确认是否新建文档。
- 仅当 `lark-doc` 不可用且用户明确同意时，才生成 `股票简称-daily-stock-analysis-lark-doc.txt` 作为可写入草稿。

飞书文档版完成后的最终回复给用户：

- 飞书文档已写入或新建完成的说明；若未写入，说明 `lark-doc` 阻塞原因。
- 日报内容中最重要的 3 条结论。
- 明确说明是否存在未获取到或替代计算的数据，以及是否完成 lark-doc 写入后的真实渲染检查。

## 数据缺失处理

如果任何字段无法获取：

- 不要编造。
- 表格缺失单元格统一写 `暂无`，避免长文本挤爆版式。
- 正文需要解释缺失时，可用短句说明“暂无可靠数据”。
- “信息来源”列出正文和表格实际引用且可打开核查的全部原始外部来源，或 `seed_finance_search（同花顺数据库）` 工具来源标识，包括公告、新闻、产业媒体、研报/预测，以及行情摘要、资金流、板块行情、技术指标、PE TTM、量比和一致预期等数据来源。缺失字段和被剔除字段不写入正文，也不写入来源列表。正文重要事实和所有展示数据必须用 `[1]`、`[2]` 形式来源编号对应到“信息来源”。不得把内部数据核验记录、本地 txt、本地材料包或中间文件作为来源。
- 若影响核心判断，降低综合判断确定性，并在风险提示中说明。

## 合规口径

本 skill 会生成研究讨论材料，不提供保证收益、确定性买卖结论或个性化交易建议。第六部分只保留“综合评估”和研究判断理由，避免使用“评级”字眼，不生成操作建议表格。

免责声明必须作为文末独立章节输出，紧跟“信息来源”。标题用 `<h1 align="left"><span text-color="orange">免责声明</span></h1>`，下方固定三段正文。固定文案见 [feishu-doc-style.md](feishu-doc-style.md) 的“免责声明”小节。

禁止：

- 承诺收益。
- 使用“必涨、稳赚、满仓”等绝对化表述。
- 引用无法验证的网络小作文作为事实依据。
