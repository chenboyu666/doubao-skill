# 视频生成

## 核心规则

当用户要求生成视频时，不要立即开始生成，也不要立即调用视频生成工具。

必须先澄清并确认以下基础参数：

- 时长：视频长度，例如 `5s`、`10s`、`15秒`。
- 当调用`text_video`比例支持：'16:9','4:3','1:1','3:4','9:16','21:9'
- 当调用`image_video`且`task_type`=`r2v`比例支持：'16:9','4:3','1:1','3:4','9:16','21:9'
- 当调用`image_video`且`task_type`=`f2v`比例仅支持保持原比例，如果用户要求调整，必须跟用户说明
- 生成详细内容：画面主体、动作、场景、风格、镜头运动、情绪氛围、文字、声音或其他限制。
- 一次最多生成2个视频，超出必须对用户进行澄清，一次仅能生成两个视频

只有在用户明确确认参数后，才可以继续生成。无论用户是否已经一次性提供完整参数，都必须先输出一次参数摘要并等待确认；不能因为信息看似完整就直接进入生成。明确确认可以是 `确认`、`可以生成`、`开始生成`、`没问题`、`yes`、`looks good` 等表达。

确认必须是**本轮任务确认**：每一个新视频任务、换产品、换 brief、换平台、换时长、换风格，都必须重新输出确认摘要并等待用户确认；历史任务的确认不得复用。`好`、`OK`、`嗯` 这类短回复只有紧跟确认摘要时才算确认；如果它们出现在新需求之后，只能视为新需求内容或继续沟通，不能直接生成。
- 默认不要让用户逐项填写所有基础参数。应先根据用户原始指令、视频用途和常见平台习惯，整理一版最优生成参数，然后只询问用户是否按这版指令生成。
- 只有当关键信息无法合理推断，或属于必须由用户提供的信息时，才追问缺失项。必须由用户提供或确认的信息包括产品图片、产品核心信息、口播台词，以
及用户明确指定但内容不完整的限制条件

## 工作流程

1. 识别用户是否有视频生成意图。
2. 判断请求属于普通视频、产品广告视频、口播/旁白视频、UGC、marketing、corporate/企业宣传商务视频，或这些类型的组合。
3. 提取用户已经给出的时长、比例、生成详细内容、产品信息、口播/旁白信息和 ref 信息。
4. 如果命中产品广告、UGC、marketing 或 corporate/企业宣传商务视频，先执行「产品/企业资料门控」。
5. 如果基础参数、产品信息或口播信息缺失或模糊，先向用户追问缺失项；如果用户已给全参数，也必须继续进入确认摘要，不能直接生成。
6. 如果包含口播、旁白或解说，还必须明确最终台词来源：用户提供、ref 提取、或由 Codex 起草；最终台词必须出现在确认摘要中。
7. 信息补齐或用户选择“无产品参考直接生成”后，向用户汇总最终视频生成参数并请求确认。
8. 如果用户修改任何参数，更新摘要并重新请求确认。
9. 用户明确确认后，才调用视频生成工具或进入视频生成流程。
10. 进入 UGC/marketing/corporate ref 策略前，先执行「分镜图片/关键帧门控」；没有明确图片需求时，只输出分镜表，不生成分镜图或关键帧图。

## 全局检索触发规则

- 用户提到“豆包”，并要求合影、互动、生成豆包形象、豆包角色/IP/头像/表情时，必须调用 `search_tool` 检索豆包 APP 角色形象；不能仅凭记忆生成豆包视觉。
- 出现时效词，如“爆火、最近、最新、新闻、今年、现在、今天”，且涉及品牌、公司、产品、IP、人物、地点、赛事、节日、热点时，必须调用 `search_tool` 获取最新信息；不能把过期记忆当作当前事实。
- 检索结果只能用于补充事实、视觉参考、平台语境和风险判断；最终生成参数仍必须回到本 MD 的澄清确认流程，未经用户确认不得直接生成。

## 检索结果使用注意点

- **人物类**：若视频中具体人物需出镜时，优先检索可靠人物图。素材和工具条件适合时，人物参考图必须作为 `person_ref/ref_images` 输入，并在确认摘要说明。
- **知识类**：已核实知识可作为 `knowledge_ref`，通过 `source_map` 保留来源，并可改写为台词/旁白；最终台词仍需用户确认。
- **混合类**：人物图与知识文本分别路由，是否采用取决于用户需求、素材可靠性和工具能力。以上均为注意点，不是生成硬门控。

## 产品/企业资料门控

命中产品广告、UGC、marketing 或 corporate/企业宣传商务视频时，默认必须先有产品上传、企业资料、品牌资产或可用 ref，再进入对应 ref 策略。

产品上传/产品 ref 包括：

- 产品图、包装图、SKU 图、商品详情截图、产品说明书、产品资料文件。
- 企业资料、官网、企业介绍、logo/IP/VI、品牌资产、产品功能介绍、办公室/工厂/门店/团队/活动素材、客户案例或服务流程资料。
- 足够明确的产品或企业文字 brief；无图片/品牌资产时只能作为概念/文字参考，必须在确认摘要中标明。

如果用户没有提供任何产品、企业资料、品牌资产或可用 ref：

1. 先澄清一次：请用户上传产品/企业/品牌参考，或明确选择“无参考直接生成/直接生成/按概念 brief 生成”。
2. 用户选择上传资料时，等待资料后再汇总确认。
3. 用户明确选择直接生成时，允许继续，但确认摘要必须写明：`参考资料：无，按概念/文字 brief 生成，产品外观、企业视觉和品牌资产不保证真实一致`。

UGC、marketing 和 corporate ref 不得绕过本门控；它们只负责在主流程确认后执行各自的分镜表、prompt 和视频生成策略。

## 分镜图片/关键帧门控

默认交付是**分镜表**，不是分镜图片。主 MD 和所有 ref 策略都必须遵守：

- 用户只说“分镜”“分镜脚本”“分镜表”“storyboard”“shooting script”时，只输出表格或文字分镜，不生成图片，不调用 imageedit/图像生成/关键帧生成工具。
- 只有用户明确说“分镜图”“分镜图片”“关键帧图”“关键帧图片”“首帧图”“每个镜头出图”“画面图”“storyboard images”“keyframe images”等图片交付词时，才可以生成分镜图片、关键帧图或首帧图。
- 如果 UGC/marketing ref 策略要求先输出分镜表，则先给分镜表；是否继续生成图片仍由本门控决定。
- 用户要最终视频但没要分镜图片时，流程是：分镜表 -> 视频 prompt -> 视频工具；不得插入分镜图生成步骤。
- 用户同时要视频和关键帧/分镜图时，流程是：分镜表 -> 关键帧/分镜图 -> 视频 prompt -> 视频工具；关键帧图数量和用途必须写入确认摘要。
- 分镜表中的台词/旁白/声音为必填字段：可以写完整原文，也可以写“无台词/无旁白，仅 BGM + 环境声”；不得缺失、留空或默认省略。确认摘要必须逐镜头列明对应台词/旁白/声音；缺少任一镜头的台词/旁白/声音确认，不得进入视频 prompt 或工具调用。
- 如果“分镜”语义不清，按分镜表处理；不要主动追问是否要图片，除非后续工具调用必须依赖图片输入。

## ref 路由策略

`ref` 是参考输入形式，不改变本 MD 的参数确认门控，也不替代用户本轮明确需求。用户提供 ref 文件、参考文件、参考图、参考视频、参考音频、参考文档、素材包或类似输入时，先判断 ref 职责，再把可用信息并入对应视频类型的参数摘要。

## 通用参考图与分镜图表策略

完整分镜图表、故事板长图、多格分镜截图或带文字说明的 storyboard，不要求裁切，也不得把整张图直接作为单一画面参考传给视频工具。此类图只能作为 `storyboard_structure_ref`：按图中阅读顺序改写为 `Shot 01 -> Shot 02 -> Shot 03...`，提取每个镜头的画面、动作、台词/旁白、重点文字、产品/人物/IP 保留点；不同镜头对应的旁白、台词，都必须转化为明确的内容和用户进行说明确认；禁止生成表格边框、编号、说明文字、截图 UI、水印。如果图中文字、顺序或镜头边界看不清，先输出拆解结果让用户确认。

所有参考图按上传、检索或传入顺序编号为 `@image1`、`@image2`、`@image3`。需要出镜的 IP、吉祥物、品牌角色、公众人物、创始人、明星、KOL 或企业高管，必须在 prompt 中明确 `@imageN` 参考什么；不强制使用 `person_ref` 或 `ip_ref` 字段名，但必须说明保留特征和防漂移约束。人物/IP 参考需保持年龄段、脸型、发型、肤色、五官比例、服装主色、轮廓比例、主色、配件、表情气质和关键识别点一致；不要换脸、换发型、换服装主色、改变体型、重绘成新角色、换颜色、拟人化或写实化。每个相关镜头都要复述核心外观锚点。只提到 IP/人物作为风格或主题灵感、且不需要出镜时，不强制进入参考图。

必须先把 ref 按职责路由：

- `product_ref`：产品图片、包装图、SKU 图、商品详情页、产品说明书、卖点表。用于补充产品图片状态、产品核心信息、包装/logo/SKU/材质/卖点；产品图是产品一致性的最高优先级参考。
- `company_ref`：官网、企业介绍、工商/年报/新闻稿、业务范围、发展阶段、团队、资质荣誉、企业公开信息。用于企业宣传、商务视频、品牌形象片和产品功能介绍中的企业事实与业务定位。
- `case_ref`：客户案例、解决方案、实施过程、结果数据、客户 logo 和授权范围。用于客户案例片、招商片、路演片和 B2B 证明链；未授权或未确认时不得生成具体客户与数据。
- `brand_ref`：品牌手册、品牌调性、品牌色、禁用词、合规要求、品牌语气。用于补充品牌限制、风格/语气、必须出现或禁止出现的信息；不得覆盖产品事实。
- `script_ref`：口播稿、旁白稿、字幕稿、直播话术、测评文案、讲解提纲。用于补充或起草口播/旁白台词；如果 ref 已有完整台词，仍要把最终台词放进确认摘要。
- `knowledge_ref`：检索并核实的事实知识；可用于台词/旁白，保留 `source_map`，未核实内容不作确定事实。
- `person_ref`：用户上传或检索的人物图；人物需出镜且条件适合时，可作为最终视频工具的 `ref_images`。
- `scene_ref`：场景图、空间图、陈列图、使用场景描述。用于补充空间、材质、光线、陈列关系；低质截图、聊天界面、杂乱桌面只提取可用产品或信息，不照搬场景。
- `style_ref`：风格图、竞品广告、参考视频、moodboard、镜头语言、调性色板。只抽象色调、光影、构图、节奏、声音氛围；不得复制具体品牌、字幕、口号、专属资产。
- `audio_ref`：配音参考、BGM、音色、节奏、情绪音频。用于补充声音语气、语速、BGM/拟音方向；不把音频内容当作画面字幕。
- `platform_ref`：平台规范、投放要求、比例、时长、禁用词、品牌合规。用于补充比例、时长、风格和限制；如与视频工具限制冲突，先说明调整并重新确认。

多 ref 同时出现时，按以下优先级合并：用户本轮明确指令 > 产品真实性、人物身份、企业事实和工具限制 > `product_ref`/`person_ref`/`company_ref` > `case_ref`/客户授权 > `knowledge_ref`/事实来源 > `script_ref`/口播台词 > `platform_ref`/品牌限制 > `brand_ref` > `scene_ref` > `style_ref`/参考案例 > `audio_ref`。冲突无法自行判断时，只追问冲突点，不重问已明确字段。

命中特定策略时，主 MD 负责视频意图识别、产品/企业资料门控、一次澄清确认和 ref 分类，不内置 UGC/marketing/corporate 详细正文；按下方策略 ref 路由读取对应文件。UGC/marketing/corporate ref 不得绕过主流程的一次确认和产品/企业资料门控；确认后再遵守被路由 ref 文件中的生成策略。

策略 ref 路由：

- UGC 口播路由：用户要口播、真人/数字人讲解、种草、测评、开箱、教程、批量达人视频、UGC 参考复刻时，读取 `references/ugc-talking-video-ref.md`。用户提到信息流、信息流带货、信息流广告、带货视频、达人带货或口播带货时，优先读取 `references/ugc-talking-video-ref.md`。
- Marketing 无分镜路由：用户要产品营销广告、商品广告、单品种草、产品展示大片、无分镜图广告片，且目标以单品消费转化、产品视觉广告、平台投放为主，并且未提信息流、带货、口播、达人或真人讲解时，读取 `references/product-marketing-ad-video-no-storyboard-ref.md`。
- Corporate 商务视频路由：用户要产品功能介绍、企业宣传片、企业产品功能宣传、商务视频、品牌形象片、企业介绍视频、招商片、路演片、会议开场片、客户案例片、雇主品牌视频、服务介绍片、B2B 广告片或公司宣传短片时，读取 `references/corporate-business-video-ref.md`。Corporate >15 秒请求必须先判断是多条独立 15 秒视频，还是完整长视频拆段拼接；拼接模式按 corporate ref 先全局规划再分段生成。
- 用户提到信息流、信息流带货、信息流广告、带货视频、达人带货或口播带货时，不走 Marketing ref；除非用户明确说不要真人/达人/口播、只要纯产品广告大片，再按确认后的目标改走 Marketing ref。
- 同时包含产品广告和真人口播时，优先看用户目标：以真人推荐/口播可信感/信息流带货为主走 UGC ref；以产品大片/广告级镜头为主且未提信息流/带货/口播时走 Marketing ref。
- Marketing 与 Corporate 冲突处理：以产品大片/广告级镜头为主且未提信息流/带货/口播时走 Marketing ref；以企业能力、公司介绍、商务信任、招商路演、客户案例、雇主品牌或会议开场为主时走 Corporate ref；如果用户同时说“品牌短片/产品宣传/功能介绍”但目标不清，只追问视频目标和受众，不直接双路由。


## 产品广告要求

当用户要求生成产品广告、商品广告视频、产品宣传片、带货视频、商业广告或类似视频时，必须在生成前额外收集：

- 产品图片：要求用户上传或提供产品图，作为视频视觉参考；也可以来自已路由为 `product_ref` 的 ref 文件。
- 产品核心信息：产品名称、核心卖点、目标用户、使用场景、风格/语气、品牌限制、必须出现的文案或行动号召；也可以从 `product_ref`、`brand_ref`、`platform_ref` 中提取。

产品广告视频必须把产品图片和产品核心信息，与基础视频参数一起合并成最终生成参数：

- 产品图片
- 产品核心信息
- ref 路由结果（如有）
- 时长
- 比例
- 生成详细内容

如果用户只提供产品想法或文字描述，但没有产品图片，必须先要求用户上传或提供产品图。除非用户明确确认要生成虚构产品或概念产品广告，否则不要自行想象产品图片。

调用视频工具前，产品广告确认摘要必须包含：

- 产品图片状态
- 产品核心信息
- ref 路由结果（如有）
- 时长
- 比例
- 生成详细内容
- 视频工具限制导致的任何必要调整

## 口播和旁白要求

当用户要求视频包含口播、旁白、解说、主播念词、真人讲解、voiceover、host speech、talking-head delivery 或类似 spoken content 时，必须在生成前明确完整台词。

口播/UGC 视频未指定时长时默认 15 秒；只有用户明确指定其他时长，或视频工具限制需要调整时，才修改默认时长并重新确认。

UGC 口播进入视频 prompt 时必须用 `{具体台词}` 包裹，只作为 `Monologue`、`audio_voiceover` 或 `Audio` 的声音/口型内容，不得作为字幕、caption、lower-third 或画面文字；重点花字/贴纸必须与口播字段分离。

Marketing 视频默认添加完整旁白台词，但旁白密度按路线决定：信息型/功能型/促销型 Marketing 视频默认 3-5 句主体旁白 + 1 句结尾 slogan；诗性/氛围/高端/香氛/珠宝/礼赠路线默认 2-3 句诗性主体旁白 + 1 句结尾 slogan。诗性旁白和 slogan 是固定搭配，不是可选关系；只有用户明确要求不要旁白或纯音乐形式时才取消旁白，其他情况均必须提供完整旁白台词。快剪 + 动感音乐/鼓点不能自动视为无旁白，除非这是用户明确指定的纯音乐/纯鼓点形式。

Marketing 确认摘要必须展示完整旁白台词、结尾 slogan 和旁白-画面落点；Marketing 旁白/口播进入视频 prompt 时必须用 `{具体台词}` 包裹，不能只写抽象旁白说明。

Corporate/企业宣传商务视频必须包含完整旁白台词，并放入确认摘要；只有用户明确要求无旁白或纯音乐时才遵从该意图，其他情况不得用一句 slogan、抽象声音描述或仅标题文案代替完整旁白。

口播台词必须具体到可以直接传给视频工具。可以要求用户提供台词，也可以根据已有产品、视频信息或 `script_ref` 先起草台词；如果由 Codex 起草，必须让用户确认或修改后才能继续生成。

调用视频工具前，口播/旁白确认摘要必须包含：

- 口播台词或旁白台词
- 口播风格、声音语气、语言、语速等用户已指定的信息
- ref 路由结果（如有）
- 时长
- 比例
- 生成详细内容
- 如果同时是产品广告，还要包含产品图片和产品核心信息
- 视频工具限制导致的任何必要调整

不要把 `加一段口播`、`做个解说`、`帮我写广告口播` 这类笼统请求视为可直接生成。最终台词必须出现在确认消息中，并由用户明确确认。

## 澄清话术

普通视频可以这样询问：

```text
我可以帮你生成视频。请先确认这三个参数：
1. 时长：
2. 比例：
3. 生成详细内容：
```

如果用户已经提供部分参数，只追问缺失或模糊的字段。

产品广告视频可以这样询问：

```text
可以。产品广告视频需要先补齐这些信息：
1. 产品图片：请上传或提供产品图，也可以提供包含产品图/卖点/品牌资料的 ref 文件
2. 产品核心信息：产品名称、核心卖点、目标用户、使用场景、风格/语气、必须出现的文案或 CTA
3. 视频参数：时长、比例、生成详细内容
```

口播/旁白视频可以这样询问：

```text
可以。包含口播/旁白的视频需要先确认台词：
1. 口播台词：请提供完整台词，或提供 script_ref/文案 ref 让我提取或起草
2. 口播风格：语气、语言、语速、声音风格（如有要求）
3. 视频参数：时长、比例、生成详细内容
```

## 确认话术

普通视频生成前必须这样汇总：

```text
请确认以下视频生成参数：
- 时长：
- 比例：
- 生成详细内容：

确认后我再开始生成。
```

产品广告视频生成前必须包含产品字段：

```text
请确认以下产品广告视频生成参数：
- 产品图片：
- 产品核心信息：
- ref 路由结果（如有）：
- 时长：
- 比例：
- 生成详细内容：

确认后我再调用视频生成工具。
```

口播/旁白视频生成前必须包含台词字段：

```text
请确认以下口播视频生成参数：
- 口播台词：
- 口播风格：
- ref 路由结果（如有）：
- 时长：
- 比例：
- 生成详细内容：

确认后我再调用视频生成工具。
```

如果视频同时是产品广告并包含口播，则确认摘要必须同时包含产品图片、产品核心信息和口播台词。

不要把沉默、部分回答、继续追问、模糊表态或无关回复当作确认。

## 生成注意事项

- 严格使用用户确认过的参数，除非目标视频工具要求格式归一化。
- 需要作为旁白/台词生成的文本内容，是否以正确的格式书写。
- 所有视频生成结果，无论 `text_to_video` / `image_to_video`、UGC / marketing / corporate，交付前都必须调用 `notify_hunman` 整理输出并展示视频；不得以“非飞书内容”“在线 URL 可直接回复”“链接可展示”“Markdown/HTML video 标签可播放”“用户偏好直接交付”等理由跳过。
- `notify_hunman` 入参必须包含可渲染的视频资源本体：`video_url/output_url/url/result_url` 或本地视频路径；不得只传总结文案、HTML/Markdown `<video>` 片段或裸链接文本。
- 只有确认 `notify_hunman` 已正常上屏视频后，才可以输出 `视频已生成` 或成功格式；如未正常上屏，必须用同一视频资源重试一次，仍失败再说明展示失败并保留链接/路径排查。
- 成功格式不得把 `video_url`、HTML/Markdown `<video>` 或裸链接作为主要交付；链接/路径仅可在 `notify_hunman` 上屏失败后的排查说明中保留。
- 如果视频工具有时长、比例、素材、口播或其他限制，先解释需要调整的地方，并让用户重新确认调整后的参数。
- 无论参数是否齐全，生成前都必须完成一次澄清确认；UGC/marketing/corporate ref 不得以“默认直接出片”为由跳过确认。
- 默认不要生成分镜图片、关键帧图或首帧图；用户未明确要求图片时，UGC/marketing/corporate 只输出分镜表并继续视频流程。
- 如果用户只是头脑风暴、写视频提示词、规划分镜或询问可用参数，正常协助即可；直到用户明确要求生成时，才进入确认流程。
- 检索后的需要出镜的人物是否作为参考图进入视频生成参数，是否在prompt中合理书写引用。
